# Todo App – Runtime & Deployment Specification

This application is a simple full-stack Todo system built with a Node.js Express backend, a PostgreSQL database, and a static frontend served from the backend. The backend runs on port 3000 and exposes a REST API that allows creating, listing, and deleting todos. It connects to a PostgreSQL instance running on hostname `postgres` at port 5432 using database name `todos_db` with credentials user `postgres` and password `devops123`. The application is designed to run in a containerized environment where PostgreSQL is reachable via Docker service networking.

The application is started using `node server.js` or via npm using `npm start`, which internally executes the same command defined in the package.json scripts section. It requires Node.js version 18 LTS or 20 LTS for stable compatibility with the dependencies used. The backend listens on port 3000 and serves both the API routes and static frontend files from the `public` directory, meaning no separate frontend server or build step is required.

The backend depends on three Node.js packages: express for handling HTTP routing and API endpoints, pg for PostgreSQL database connectivity and query execution, and cors to enable cross-origin requests between the frontend and backend during development or containerized execution. These dependencies are installed using `npm install` before starting the application.

The application does not rely on environment variables in its current implementation; instead, database configuration is hardcoded directly into the PostgreSQL Pool initialization. However, in a production-grade setup, these values would typically be externalized into environment variables such as DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD, and PORT for flexibility and security.

Database connectivity is handled using a PostgreSQL connection pool that connects to host `postgres` on port 5432 and targets the database `todos_db`. The equivalent connection string representation is `postgresql://postgres:devops123@postgres:5432/todos_db`, although the application uses object-based configuration instead of a URL string. All SQL execution is performed through parameterized queries to avoid injection risks.

The database schema is not embedded permanently in the application logic; instead, it is defined in an external initialization file located at `db/init.sql`. This file contains a single table definition for `todos`, where each todo has an auto-incrementing primary key `id`, a required text field `title`, and a timestamp field `created_at` which defaults to the current time. This SQL file is intended to be executed automatically by PostgreSQL during container initialization when mounted into `/docker-entrypoint-initdb.d/`, and it is only executed once when the database volume is first created. If the database volume already exists, the script will not rerun.

The required project structure consists of a root application directory containing server.js and package.json, a db directory containing init.sql for schema initialization, and a public directory containing index.html which serves the frontend UI. The frontend is a simple HTML-based interface that uses JavaScript fetch calls to communicate with the backend API. It allows users to submit new todos through a form, retrieve and display all todos from the backend, and delete individual todos using a delete button next to each item.

At runtime, the full application requires only that the Node dependencies are installed and the server is started. There is no build step, no bundling process, and no compilation required. The startup sequence is therefore strictly `npm install` followed by `node server.js`. In containerized environments, the runtime must ensure that the backend service waits for PostgreSQL availability before executing queries, otherwise connection attempts may fail during initial startup.

For deployment scenarios, the application expects the following runtime components to be available: server.js, package.json, the db directory containing init.sql, and the public directory containing frontend assets. Node modules are installed inside the runtime environment and are not expected to be copied manually. In Docker-based deployments, restart policies such as `unless-stopped` or `always` are recommended to ensure the application automatically recovers from unexpected crashes.

If the application crashes during execution, the default behavior is that the Node process terminates immediately without recovery. In production environments, process managers such as PM2 or container orchestrators like Docker Compose or Kubernetes should be used to ensure automatic restarts and improved resilience.

Overall, this application is intentionally minimal and serves as a learning-grade full-stack system demonstrating REST API design, PostgreSQL integration, and basic frontend-backend interaction without build tooling or complex infrastructure layers.
